## 1. 系統概述 (System Overview)

### 1.1 產品目標
本產品旨在解決小學資訊老師備課時「尋找合適科技新聞素材耗時且品質不一」的痛點。透過自動化彙整與預先篩選，提供一個專注於「即時、適合小學教學」的新聞清單，協助老師在 1 分鐘內找到可用案例。

### 1.2 核心價值主張
*   **省時**: 免去 Google 搜尋與篩選的時間。
*   **適用**: 內容經過篩選，確保適合小學生（資訊素養、AI、數位生活）。
*   **穩定**: 固定頻率更新，確保隨時有新素材。

## 2. 系統架構分析 (System Architecture Analysis)

基於 MVP (Minimum Viable Product) 的「快」與「純粹」原則，系統採用 **單向資訊流** 架構。

### 2.1 高階架構圖
```mermaid
graph LR
    A[外部新聞來源] -->|排程抓取| B(爬蟲與資料處理模組)
    B -->|適合度過濾| C(資料庫 / 資料儲存)
    C -->|讀取清單| D[前端展示頁面]
    D -->|點擊連結| E[使用者 (老師)]
```

### 2.2 關鍵模組分析

1.  **資料採集層 (Data Collection Layer)**
    *   **目標**: 從指定來源獲取新聞。
    *   **來源清單 (MVP)**:
        *   教育部/教育科技官網
        *   TechNews (科技新報)
        *   數位時代 (Business Next)
        *   泛科學 (PanSci) - 科技類別
        *   公視新聞網
    *   **策略**: 每週一上午 8:00 執行一次排程 (Cron Job)。

2.  **資料處理與篩選層 (Processing & Filtering Layer)**
    *   **目標**: 確保新聞適合小學生。
    *   **MVP 篩選邏輯**:
        *   **白名單關鍵字**: "AI", "機器人", "太空", "環保", "資訊安全", "程式教育", "發明".
        *   **黑名單關鍵字 (排除)**: "股市", "營收", "併購", "腥羶色", "政治", "深奧技術(如半導體製程細節)".
        *   *(註: MVP 階段若自動化準確度不足，可保留一個簡單的「人工勾選」後台介面，由管理員每週花 10 分鐘快速勾選)*。

3.  **展示層 (Presentation Layer)**
    *   **目標**: 極簡、快速載入。
    *   **形式**: 單頁式應用 (Single Page Application) 或 靜態網站 (Static Site)。
    *   **互動**: 純瀏覽，無登入，無複雜互動。

## 3. 功能規格 (Functional Specifications)

### 3.1 P0: 自動化新聞彙整 (Automated Aggregation)
*   **F1.1 來源管理**: 系統需支援設定多個 RSS 源或網頁爬取規則。
*   **F1.2 排程更新**: 系統需在每週指定時間自動觸發更新流程。
*   **F1.3 資料解析**: 需解析出 標題 (Title)、原文連結 (URL)、發布日期 (Date)、來源名稱 (Source)。

### 3.2 P0: 適用性篩選 (Educational Filtering)
*   **F2.1 關鍵字過濾**: 依據預設的教育關鍵字清單進行初步篩選。
*   **F2.2 時效性過濾**: 僅保留最近 3 個月內的新聞，過舊新聞自動封存或不顯示(保持清單新鮮)。

### 3.3 P0: 前端新聞清單 (News Feed UI)
*   **F3.1 清單列表**: 依日期排序（新 -> 舊）。
*   **F3.2 新聞卡片**: 每則新聞顯示：
    *   新聞標題 (點擊開新視窗跳轉原始網頁)
    *   來源網站標籤 (如：[科技新報])
    *   發布日期 (如：2023/10/25)
*   **F3.3 載入效能**: 首屏載入時間 < 2秒。

## 4. 資料庫設計 (Data Schema)

雖然 MVP 不做複雜功能，但需定義基礎資料結構以便擴充。

**Table: NewsItem**

| 欄位名稱 | 類型 | 說明 |
| :--- | :--- | :--- |
| id | UUID | 唯一識別碼 |
| title | String | 新聞標題 |
| url | String | 原始新聞連結 |
| source_name | String | 來源網站名稱 (e.g., "TechNews") |
| published_at | DateTime | 原始發布時間 |
| created_at | DateTime | 系統抓取時間 |
| is_visible | Boolean | 是否顯示 (預設 True，若需下架可改 False) |
| tags | Array<String> | (保留欄位) 自動標記的關鍵字，如 ["AI", "資安"] |

## 5. 使用者介面與體驗設計 (UI/UX Design)

### 5.1 介面佈局 (Wireframe Concept)
*   **Header**:
    *   標題：「小學科技新聞週報」 (或是更活潑的名稱)
    *   副標：「每週更新，專為老師備課精選」
    *   包含「本週最新」標籤，顯示最後更新日期。
*   **Main Content**:
    *   **列表區**: 垂直條列式卡片。
    *   **卡片樣式**:
        *   左側/上方：新聞標題 (字體大清楚)。
        *   下方輔助資訊：來源 ICON + 日期。
        *   *視覺提示*：如果是本週新抓取的新聞，顯示 "NEW" 標籤。
*   **Footer**:
    *   簡單版權聲明。
    *   回報連結 ("發現這則新聞不適合？") - 連接 Google Form 即可。

### 5.2 互動流程
1.  老師打開網頁。
2.  看到最新日期的標題。
3.  瀏覽標題，看到感興趣的關鍵字 (如 "小學生也能懂的 ChatGPT...")。
4.  點擊標題 -> 新分頁開啟原始新聞頁面。
5.  老師閱讀後決定是否使用。

## 6. 技術可行性與限制 (Feasibility & Constraints)

### 6.1 爬蟲限制
*   需注意各新聞網站的 robots.txt 規範。
*   若網站結構頻繁變動，爬蟲可能會失效 (需有錯誤通知機制，寄信給開發者)。

### 6.2 版權聲明
*   本產品**不轉載全文**，也不抓取圖片 (避免版權爭議)，僅提供「標題」與「外部連結」。
*   此方式符合「超連結」的網路慣例，法律風險最低。

### 6.3 成本估算
*   **Compute**: 使用免費層級的雲端函數 (AWS Lambda / Vercel Functions / Github Actions) 執行每週排程。
*   **Database**: 輕量級 NoSQL 或甚至 JSON 檔案 (若資料量少，Git-based CMS 即可)。
*   **Hosting**: Vercel / Netlify / Github Pages (免費)。
*   **總結**: MVP 營運成本趨近於零。
